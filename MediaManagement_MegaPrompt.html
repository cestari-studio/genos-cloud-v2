Media Management System

Análise detalhada da solicitação + prompts melhorados para implementação completa do sistema de mídias do genOS Lumina.

⚡ REGRA DE OURO — CARBON ABSOLUTA

100% @carbon/react + @carbon/ibm-products + @carbon/icons-react. ZERO Tailwind. ZERO inline styles hardcoded. AILabel em toda feature IA. Upload usa FileUploader / FileUploaderDropContainer do Carbon.

📋 Análise da Solicitação Original

O que foi pedido:

  • Tenant Selector: Master e Agency veem dropdown para filtrar posts por tenant filho (incluindo "Todos").
  • Upload Media: Ao expandir post → Visualizar → botão "Upload Media" abre novo modal.
  • Modal Step 1: Upload de mídias (imagens e/ou vídeos) relacionadas ao post.
  • Modal Step 2: Após upload, atribuir cada mídia ao card/slot correto via dropdown (itens = quantidade exata de cards do post).
  • Auto-Rename: Ao salvar, sistema renomeia automaticamente: {TenantName} - {DataPostagem} - {PostId} - {Sequência}.{ext}
  • Cascading Updates: Se data do post ou nome do tenant mudar → renomear todas as mídias vinculadas automaticamente.

O que JÁ existe no codebase:

  • ✅ Tabela post_media (id, post_id, position, type, wix_media_url, file_name, status)
  • ✅ Bucket content-media no Supabase Storage
  • ✅ Edge Function assign_media com lógica de rename parcial
  • ✅ CarouselPreview component que renderiza mídias por position
  • ✅ buildMediaRefMap() no MatrixList que indexa mídias por postId
  • ✅ RLS baseada em tenant hierarchy

🔧 O que Falta + Melhorias que Adicionei

Gaps da solicitação original:

  • ❌ Validação de formato de mídia: Quais formatos aceitos? Tamanho máximo? Dimensões recomendadas por formato (feed vs stories vs reels)?
  • ❌ Feedback visual de upload: Progress bar individual por arquivo? Tratamento de erro por arquivo?
  • ❌ Substituição de mídia: E se o user quiser trocar uma mídia já atribuída? Precisa deletar e re-upload ou pode sobrescrever?
  • ❌ Preview de mídia: Thumbnail após upload? Preview de vídeo?
  • ❌ Feed e Reels: Para Feed (1 mídia) e Reels (1 vídeo), o dropdown de atribuição é desnecessário — atribuir automaticamente.
  • ❌ Stories: Stories pode ter múltiplas mídias (stories sequenciais) — tratar como carrossel?
  • ❌ Limites: Máximo de mídias por post? Tamanho total máximo?
  • ❌ Permissão de Client: Client tenant pode fazer upload? Ou só Master/Agency?
  • ❌ Ordem de atribuição: E se o user atribuir 2 mídias ao mesmo card? Validação?

Melhorias adicionadas ao prompt:

  • ✅ Validação por formato: Feed = 1 image/video, Carrossel = até 10 image/video, Stories = até 10 image/video, Reels = 1 video only
  • ✅ Auto-assign para Feed/Reels: Se formato = feed ou reels, pular step 2 e atribuir position=1 automaticamente
  • ✅ Thumbnail preview: Após upload, mostrar thumbnail (image) ou primeiro frame (video) com nome e tamanho
  • ✅ Drag & drop: Carbon FileUploaderDropContainer para drag & drop + click to browse
  • ✅ Replace/Delete individual: Cada mídia atribuída pode ser substituída ou removida individualmente
  • ✅ Validação de duplicata: Não permitir 2 mídias no mesmo position (dropdown desabilita opções já atribuídas)
  • ✅ Trigger SQL para cascade rename: Ao alterar scheduled_date ou tenant name, trigger renomeia no storage + atualiza post_media.file_name
  • ✅ Client pode upload: Client com post em status "draft" ou "revision_requested" pode fazer upload. Master/Agency podem em qualquer status.
  • ✅ Supabase Storage RLS: Policies por tenant_id no bucket path
  • ✅ Metadata extraction: Extrair dimensões e duração via Edge Function após upload

-- 2. Função de nomenclatura padrão
-- ═══════════════════════════════════════

CREATE OR REPLACE FUNCTION generate_media_filename(
  p_tenant_name TEXT,
  p_scheduled_date TIMESTAMPTZ,
  p_post_id UUID,
  p_position INTEGER,
  p_extension TEXT
) RETURNS TEXT AS $$
DECLARE
  v_clean_name TEXT;
  v_date_str TEXT;
  v_post_short TEXT;
BEGIN
  -- Sanitizar nome do tenant (remover espaços, acentos, caracteres especiais)
  v_clean_name := regexp_replace(
    unaccent(p_tenant_name),
    '[^a-zA-Z0-9]', '', 'g'
  );
  
  -- Formatar data (usar created_at do post se scheduled_date for NULL)
  v_date_str := to_char(COALESCE(p_scheduled_date, now()), 'YYYY-MM-DD');
  
  -- Primeiros 8 chars do UUID do post
  v_post_short := left(p_post_id::text, 8);
  
  -- Formato final: TenantName-2026-03-15-a1b2c3d4-01.jpg
  RETURN v_clean_name || '-' || v_date_str || '-' || v_post_short || '-' || 
         lpad(p_position::text, 2, '0') || '.' || lower(p_extension);
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- ═══════════════════════════════════════
-- 3. Trigger: Rename mídias ao alterar scheduled_date do post
-- ═══════════════════════════════════════

CREATE OR REPLACE FUNCTION trigger_rename_media_on_post_update()
RETURNS TRIGGER AS $$
DECLARE
  v_media RECORD;
  v_tenant_name TEXT;
  v_new_filename TEXT;
  v_new_path TEXT;
  v_ext TEXT;
BEGIN
  -- Só executa se scheduled_date mudou
  IF OLD.scheduled_date IS DISTINCT FROM NEW.scheduled_date THEN
    -- Buscar nome do tenant
    SELECT name INTO v_tenant_name FROM tenants WHERE id = NEW.tenant_id;
    
    -- Para cada mídia do post
    FOR v_media IN 
      SELECT * FROM post_media WHERE post_id = NEW.id AND storage_path IS NOT NULL
    LOOP
      -- Extrair extensão
      v_ext := split_part(v_media.file_name, '.', -1);
      
      -- Gerar novo nome
      v_new_filename := generate_media_filename(
        v_tenant_name, NEW.scheduled_date, NEW.id, v_media.position, v_ext
      );
      
      -- Novo path no storage
      v_new_path := NEW.tenant_id || '/' || NEW.id || '/' || v_new_filename;
      
      -- Atualizar registro (o move no Storage será feito pela Edge Function)
      UPDATE post_media 
      SET file_name = v_new_filename,
          storage_path = v_new_path
      WHERE id = v_media.id;
    END LOOP;
    
    -- Criar evento para Edge Function processar os moves no Storage
    INSERT INTO activity_log (tenant_id, category, action, details)
    VALUES (NEW.tenant_id, 'media', 'rename_required', jsonb_build_object(
      'post_id', NEW.id,
      'reason', 'scheduled_date_changed',
      'old_date', OLD.scheduled_date,
      'new_date', NEW.scheduled_date
    ));
  END IF;
  
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_rename_media_on_post_update
  AFTER UPDATE ON posts
  FOR EACH ROW
  WHEN (OLD.scheduled_date IS DISTINCT FROM NEW.scheduled_date)
  EXECUTE FUNCTION trigger_rename_media_on_post_update();

-- ═══════════════════════════════════════
-- 4. Trigger: Rename mídias ao alterar nome do tenant
-- ═══════════════════════════════════════

CREATE OR REPLACE FUNCTION trigger_rename_media_on_tenant_update()
RETURNS TRIGGER AS $$
BEGIN
  IF OLD.name IS DISTINCT FROM NEW.name THEN
    -- Criar evento para Edge Function processar batch rename
    INSERT INTO activity_log (tenant_id, category, action, details)
    VALUES (NEW.id, 'media', 'batch_rename_required', jsonb_build_object(
      'reason', 'tenant_name_changed',
      'old_name', OLD.name,
      'new_name', NEW.name
    ));
    
    -- Atualizar file_name de todas as mídias de todos os posts deste tenant
    UPDATE post_media pm
    SET file_name = generate_media_filename(
      NEW.name,
      p.scheduled_date,
      p.id,
      pm.position,
      split_part(pm.file_name, '.', -1)
    ),
    storage_path = p.tenant_id || '/' || p.id || '/' || generate_media_filename(
      NEW.name,
      p.scheduled_date,
      p.id,
      pm.position,
      split_part(pm.file_name, '.', -1)
    )
    FROM posts p
    WHERE pm.post_id = p.id AND p.tenant_id = NEW.id AND pm.storage_path IS NOT NULL;
  END IF;
  
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_rename_media_on_tenant_update
  AFTER UPDATE ON tenants
  FOR EACH ROW
  WHEN (OLD.name IS DISTINCT FROM NEW.name)
  EXECUTE FUNCTION trigger_rename_media_on_tenant_update();

-- 5. RLS para post_media
-- ═══════════════════════════════════════

ALTER TABLE post_media ENABLE ROW LEVEL SECURITY;

-- Master: vê tudo
CREATE POLICY post_media_master_all ON post_media
  FOR ALL TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM posts p
      JOIN tenant_members tm ON tm.tenant_id IN (
        SELECT id FROM tenants WHERE depth_level = 0
      )
      WHERE p.id = post_media.post_id
      AND tm.user_id = auth.uid()
    )
  );

-- Agency: vê próprio + filhos
CREATE POLICY post_media_agency ON post_media
  FOR ALL TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM posts p
      JOIN tenants t ON p.tenant_id = t.id
      JOIN tenant_members tm ON tm.user_id = auth.uid()
      WHERE p.id = post_media.post_id
      AND (t.id = tm.tenant_id OR t.parent_tenant_id = tm.tenant_id)
    )
  );

-- Client: vê próprio
CREATE POLICY post_media_client ON post_media
  FOR ALL TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM posts p
      JOIN tenant_members tm ON tm.tenant_id = p.tenant_id
      WHERE p.id = post_media.post_id
      AND tm.user_id = auth.uid()
    )
  );

P2 — Edge Function: media-manager

REGRA: Edge Function Deno/TypeScript com Supabase client.

TAREFA: Criar supabase/functions/media-manager/index.ts para gerenciar todo o ciclo de vida de mídias.

AÇÕES:

1. "upload" — Upload de arquivo para Storage temporário
   Input: FormData com files[] + { post_id, tenant_id }
   Fluxo:
   a) Validar permissão (user é membro do tenant ou parent)
   b) Validar formato: accept = ['image/jpeg', 'image/png', 'image/webp', 'image/heic', 'video/mp4', 'video/quicktime']
   c) Validar tamanho: max 50MB por arquivo, max 200MB total por post
   d) Upload para temp path: content-media/{tenantId}/temp/{postId}/{uuid}.{ext}
   e) Criar registro em post_media com status='pending', original_file_name=nome original
   f) Se imagem: extrair width/height via sharp (ou metadata)
   g) Se vídeo: extrair duration_seconds via ffprobe (se disponível) ou armazenar null
   h) Retornar: { success, media: { id, original_file_name, type, file_size, width, height, duration_seconds, thumbnail_url } }

2. "assign" — Atribuir mídias aos positions e renomear
   Input: { post_id, tenant_id, assignments: [{ media_id, position }] }
   Fluxo:
   a) Validar que positions são válidos (1 a card_data.length ou 1 para feed/reels)
   b) Validar que nenhum position está duplicado
   c) Para cada assignment:
      - Buscar post (tenant_name via JOIN, scheduled_date)
      - Gerar filename via nomenclatura: {TenantName}-{Date}-{PostId8}-{Position}.{ext}
      - Move no Storage: temp path → final path ({tenantId}/{postId}/{filename})
      - Atualizar post_media: position, file_name, storage_path, wix_media_url (public URL), status='uploaded'
   d) Retornar: { success, media: [...updated media records] }

3. "delete" — Remover mídia individual
   Input: { media_id, post_id }
   Fluxo:
   a) Validar permissão
   b) Deletar arquivo do Storage
   c) Deletar registro de post_media
   d) Retornar: { success }

4. "replace" — Substituir mídia existente
   Input: FormData com file + { media_id, post_id }
   Fluxo:
   a) Deletar arquivo antigo do Storage
   b) Upload novo arquivo
   c) Gerar novo filename (mantendo position)
   d) Move para path final
   e) Atualizar post_media (file_name, wix_media_url, file_size, mime_type, width, height)
   f) Retornar: { success, media: updated record }

5. "process_renames" — Processar renames pendentes (chamado por CRON ou webhook)
   Input: { post_id } ou { tenant_id, reason }
   Fluxo:
   a) Buscar post_media onde storage_path != path esperado (comparar com generate_media_filename)
   b) Para cada mídia desatualizada:
      - storage.from('content-media').move(old_path, new_path)
      - Atualizar wix_media_url com nova public URL
   c) Retornar: { success, renamed_count }

TAREFA: Implementar o processamento real de renames no Storage (os triggers SQL atualizam post_media.file_name e storage_path, mas o arquivo físico no bucket precisa ser movido pela Edge Function).

PASSO 1 — EDGE FUNCTION: media-rename-processor

Criar supabase/functions/media-rename-processor/index.ts:

Pode ser chamado de 2 formas:
A) Via webhook quando activity_log recebe category='media', action='rename_required' ou 'batch_rename_required'
B) Via CRON a cada 5 minutos para processar pendências

Fluxo:
1. Buscar activity_log com category='media' AND action IN ('rename_required', 'batch_rename_required') AND processed = false
   (adicionar coluna 'processed BOOLEAN DEFAULT false' no activity_log se não existir)
   
2. Para cada evento:
   a) Se action='rename_required' (post date mudou):
      - Buscar post_media WHERE post_id = details->>'post_id'
      - Para cada mídia:
        - Old path: extrair de wix_media_url (tudo após /content-media/)
        - New path: storage_path (já atualizado pelo trigger)
        - Se old_path != new_path:
          → storage.from('content-media').move(old_path, new_path)
          → Atualizar wix_media_url com nova public URL
      - Marcar activity_log.processed = true
      
   b) Se action='batch_rename_required' (tenant name mudou):
      - Buscar TODOS os post_media de posts desse tenant que têm storage_path atualizado pelo trigger
      - Processar em batch (máx 50 por execução para não exceder timeout)
      - Move cada arquivo
      - Atualizar wix_media_url
      - Marcar processed = true

3. Logging:
   - Para cada rename bem-sucedido: log no console
   - Para erros: registrar em activity_log com category='media', action='rename_error'

PASSO 2 — MODIFICAR update_status na content-factory-ai:

Quando scheduled_date é alterada via update_status:
- O trigger SQL já cuida de atualizar post_media
- Criar activity_log entry (o trigger já faz isso)
- O processor irá mover os arquivos

PASSO 3 — TESTE DE CASCADING:

Cenário 1: Mudar scheduled_date de um post com 3 mídias
→ Trigger atualiza file_name e storage_path de 3 registros
→ Activity_log criado com 'rename_required'
